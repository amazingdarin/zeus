# 文档块级背景色与字体色功能设计

日期：2026-03-01  
状态：已确认（可进入实施计划）

## 1. 目标与范围

为文档编辑器的文本类块增加统一样式能力：

1. 支持设置块背景色。
2. 支持设置块级字体色（对整个块文本生效）。
3. 颜色来源仅允许预设色板，不支持自定义输入。

本期范围仅包含文本类块，不覆盖图片、图表、插件块与代码块。

## 2. 已确认产品规则

1. 适用块类型：
   - `paragraph`、`heading`、`blockquote`
   - `bulletList`、`orderedList`、`taskList`
   - `listItem`、`taskItem`
   - `tableCell`、`tableHeader`
2. 颜色策略：
   - 仅预设颜色（受控 token）
   - 背景色与文字色分别独立设置/清除
3. 字体色语义：
   - 采用块级默认字体色，不做行内局部文本色
4. 交互入口：
   - 放在块操作菜单（`⋮⋮`）内，不复用顶部行内高亮按钮

## 3. 方案对比与决策

### 方案A：块级属性方案（采用）

通过节点 `attrs` 持久化 `backgroundColor` 和 `textColor`，由扩展统一解析、渲染与命令操作。

优点：
1. 与“块级样式”语义完全一致。
2. 对多块选择/表格单元批量操作天然友好。
3. 数据模型清晰，回放与持久化稳定。

缺点：
1. 需扩展节点全局属性与命令集。

### 方案B：行内 mark 伪块级

通过 `highlight` + `Color/TextStyle` mark 将样式铺满块文本。

优点：可复用 mark 生态。  
缺点：语义偏差明显，列表/嵌套结构一致性差，维护成本高。

### 方案C：键值 class 映射方案

JSON 存 `bgKey/textKey`，运行时映射 class。

优点：主题控制强。  
缺点：解析映射层增加，不必要复杂。

结论：采用方案A。

## 4. 架构与数据模型设计

### 4.1 节点属性

在目标块节点追加属性：

1. `backgroundColor?: string | null`
2. `textColor?: string | null`

颜色值仅允许受控 token，例如：
1. `var(--tt-color-highlight-yellow)`
2. `var(--tt-color-text-blue)`

### 4.2 扩展实现

新增统一扩展（建议：`NodeBlockStyleExtension`）：

1. `addGlobalAttributes` 注入 `backgroundColor`、`textColor`
2. `parseHTML` 从 `style.backgroundColor` 与 `style.color` 回读
3. `renderHTML` 输出对应 style
4. 命令能力：
   - `setNodeBackgroundColor(color)`
   - `unsetNodeBackgroundColor()`
   - `setNodeTextColor(color)`
   - `unsetNodeTextColor()`
5. 命令仅对白名单块类型生效

兼容策略：
1. 保留当前 `nodeBackground` 相关命令入口（可做 alias）。
2. 旧文档无新属性时按默认样式渲染。

### 4.3 OpenSpec 更新

1. 更新 `openspec/specs/document-format/specs/block-types.spec.md`  
为相关块类型新增 `backgroundColor` / `textColor` 可选属性说明与示例。
2. `marks.spec.md` 不变（本期不引入行内文本色 mark）。

## 5. 交互与可用性设计

### 5.1 块菜单结构

在块操作菜单新增：

1. `块背景色`（二级色板）
2. `块文字色`（二级色板）
3. 各自包含 `清除颜色` 操作

### 5.2 操作行为

1. 默认作用于当前光标所在块。
2. 表格/多块选择时按已选节点批量应用。
3. 点击颜色为幂等设置，不做反向 toggle。
4. 清除行为独立按钮执行。

### 5.3 状态反馈

1. 当前颜色显示选中态。
2. 多块颜色不一致时显示“混合态”（不高亮具体色块，但允许覆盖写入）。

## 6. 数据流与兼容性

### 6.1 数据流

1. 用户在块菜单选色。
2. 编辑器命令更新块 `attrs`。
3. `onChange` 输出 JSON。
4. 复用现有保存链路持久化。

### 6.2 兼容性

1. 向后兼容：旧文档可正常加载。
2. 主题兼容：颜色 token 随主题变量自动切换。
3. 导出降级：若 markdown/docx 转换链路不支持块级颜色，允许忽略该样式并在文档中注明降级行为。

## 7. 安全与约束

1. 命令层校验颜色值必须来自预设 token 白名单。
2. 非白名单节点不执行样式写入（返回 `false`）。
3. 解析未知颜色值时可保留或清洗，首版推荐清洗为 `null` 以保证受控输出。

## 8. 测试策略

### 8.1 编辑器扩展单测（`packages/doc-editor`）

1. 单块设置/清除背景色。
2. 单块设置/清除文字色。
3. 多块批量应用成功。
4. 非法颜色被拒绝。
5. 非目标节点不生效。
6. parse/render 往返一致。

### 8.2 Web 集成测试

1. 菜单可见“块背景色/块文字色”。
2. 应用后切换文档再返回样式保留。
3. 清除后样式恢复默认。

### 8.3 Playwright 回归

1. 设置背景色/文字色后刷新页面仍保持。
2. 多页签切换与文档树切换不丢样式。

## 9. 风险与回滚

1. 风险：
   - 列表与嵌套块命中范围不一致
   - 表格单元批量设置边界行为复杂
2. 回滚：
   - 快速隐藏 UI 入口
   - 移除扩展注册停止新写入
   - 历史 JSON 中额外 attrs 可被安全忽略

## 10. 非目标（首版不做）

1. 行内局部文本色。
2. 自定义颜色拾取器。
3. 图片/图表/插件块颜色样式。
4. 代码块颜色定制。

## 11. 结论

首版采用“块级属性 + 受控色板 + 块菜单入口”的最小闭环方案，在保证语义正确与可维护性的前提下交付稳定能力，并为后续移动端入口与高级配色策略预留扩展空间。
