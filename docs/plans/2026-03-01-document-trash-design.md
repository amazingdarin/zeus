# 文档垃圾箱功能设计

## 1. 背景与目标

当前文档删除为物理硬删除，误删不可恢复，不满足日常文档协作场景下的安全删除需求。
本设计引入项目级垃圾箱（Recycle Bin）能力，实现“先回收、后清理”的删除模型，并支持自动清理策略。

核心目标：
- 文档/目录删除默认进入垃圾箱，不直接硬删除。
- 提供单条恢复、单条彻底删除、全量清空垃圾箱。
- 支持可配置自动清理（默认关闭，建议 30 天）。
- 保持与多端同步、知识索引一致性兼容。

## 2. 产品行为与规则

### 2.1 删除行为
- 删除文档或目录时，统一执行“移入垃圾箱”。
- 目录删除按“单条垃圾箱记录”处理，记录整个子树，不拆分子项。

### 2.2 恢复行为
- 单条恢复回原位置。
- 若原父级已不存在：自动恢复到文档根目录。
- 若目标位置名称冲突：自动重命名，如 `原标题（恢复）`、`原标题（恢复2）`。

### 2.3 清理行为
- 支持单条彻底删除。
- 支持清空垃圾箱（全量彻底删除）。
- 本期不做批量多选操作。

### 2.4 自动清理策略
- 默认关闭自动清理。
- 用户可在通用设置中开启并配置保留天数（默认值 30）。
- 开启后由后端定时任务清理超过阈值的垃圾项。

### 2.5 真相源与同步语义
- 与当前文档同步策略保持一致，不额外引入新真相源切换机制。
- 已删除文档在垃圾箱内视为“软删除态”，对正常文档树不可见。

## 3. 后端模型与 API 设计

### 3.1 存储模型（文件系统方案）
按项目作用域在 `docs/` 下新增垃圾箱目录：

- `docs/.trash/entries/<trashId>/...`：存放被删除文档或目录实体。
- `docs/.trash/index.json`：垃圾箱索引元数据。

索引项建议字段：
- `trashId`
- `entityType`（`document` | `directory`）
- `title`
- `originalPath`
- `deletedAt`
- `deletedBy`
- `parentIdSnapshot`（用于恢复参考）
- `sizeMeta`（可选：字节数/子项数量）

### 3.2 删除语义改造
- 现有删除流程从“物理删除”改为“move to trash”。
- 写入顺序：移动文件到 `entries`，更新 `index.json`。
- 删除后触发知识索引移除流程。

### 3.3 API（owner-scope 路由）
在项目作用域下新增垃圾箱接口：

- `GET /api/projects/:ownerType/:ownerKey/:projectKey/trash`
  - 查询垃圾箱列表。
- `POST /api/projects/:ownerType/:ownerKey/:projectKey/trash/:trashId/restore`
  - 恢复单条记录。
- `DELETE /api/projects/:ownerType/:ownerKey/:projectKey/trash/:trashId`
  - 彻底删除单条记录。
- `DELETE /api/projects/:ownerType/:ownerKey/:projectKey/trash`
  - 清空垃圾箱。

### 3.4 设置扩展
复用 `general settings`：
- `trashAutoCleanupEnabled: boolean`
- `trashAutoCleanupDays: number`

由 `GET/PUT /api/settings/general` 统一读写。

### 3.5 自动清理任务
- 在 app-backend 启动后挂载周期性任务（例如每小时扫描）。
- 仅在 `trashAutoCleanupEnabled=true` 时执行。
- 清理前后记录日志并处理部分失败重试。

## 4. 前端信息架构与交互

### 4.1 入口
- 文档树增加“垃圾箱”入口（项目级）。
- 文档删除按钮文案改为“移入垃圾箱”。

### 4.2 垃圾箱视图
列表字段：
- 标题
- 类型（文档/目录）
- 原位置
- 删除时间
- 大小/子项数（目录）
- 操作

空状态：`垃圾箱为空`。

### 4.3 单条操作
- `恢复`
- `彻底删除`（二次确认）

操作后提示明确恢复结果（如恢复到根目录、自动重命名）。

### 4.4 全量操作
- 顶部提供 `清空垃圾箱`。
- 强确认（例如二次确认或输入确认词）后执行。

### 4.5 设置 UI
通用配置增加垃圾箱自动清理项：
- 开关：启用自动清理（默认关）。
- 数值：保留天数（默认 30，建议范围 1-3650）。

## 5. 可靠性与一致性

### 5.1 索引一致性
- `index.json` 写入采用临时文件 + 原子重命名。
- 服务启动时进行垃圾箱索引自检与修复：
  - 目录存在但索引缺失 -> 重建索引项（可降级标记）。
  - 索引存在但目录丢失 -> 清理脏记录。

### 5.2 幂等语义
- 恢复、单条彻删、清空接口均应幂等。
- 重复请求返回可收敛结果，避免前端异常。

### 5.3 并发控制
- 同一项目垃圾箱操作加互斥锁（project scope 级别）。
- 清空操作与单条操作互斥，防止并发写索引冲突。

### 5.4 失败补偿
- 关键流程出现中间失败时记录补偿任务/重试点。
- 恢复或清理后同步知识索引状态，避免检索脏数据。

### 5.5 观测性
记录结构化日志字段：
- `projectScope`
- `trashId`
- `operation`
- `durationMs`
- `result`
- `errorCode`

## 6. 测试策略与验收标准

### 6.1 后端测试
- 删除入箱（文档/目录）
- 恢复（原路径、父级缺失、重名冲突）
- 单条彻删与清空
- 自动清理开关与阈值
- 接口幂等与并发安全

### 6.2 API 集成测试
- `GET /trash`
- `POST /trash/:id/restore`
- `DELETE /trash/:id`
- `DELETE /trash`
- 通用设置中的垃圾箱字段读写

### 6.3 前端自动化（Playwright）
- 删除后进入垃圾箱
- 恢复/彻删/清空流程
- 设置页开启自动清理并保存
- 恢复冲突提示文案

### 6.4 验收标准（DoD）
- 删除默认入箱，不直接硬删。
- 支持恢复、单条彻删、清空。
- 自动清理默认关闭，开启后按天数生效。
- 冲突恢复自动回退且可见提示。
- 关键流程测试通过并稳定。

## 7. 范围边界（本期不做）
- 垃圾箱批量多选/批处理。
- 垃圾箱全文检索与高级筛选。
- 回收站历史审计视图（可后续扩展）。
