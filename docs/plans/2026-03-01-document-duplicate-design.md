# 文档与块“创建副本”功能设计

日期：2026-03-01  
状态：已确认（可进入实施计划）

## 1. 目标与范围

为文档系统增加“创建副本”能力，覆盖两类场景：

1. 文档级副本：在文档功能中创建当前文档的副本。
2. 块级副本：在块菜单中创建当前块的副本。

本次仅做首版可用能力，遵循最小闭环与低风险原则。

## 2. 已确认产品规则

### 2.1 文档级副本

1. 复制范围：仅复制当前文档本体（标题+内容），不复制子文档。
2. 创建位置：与原文档同父级。
3. 命名规则：`原标题（副本）`；若重名递增为 `原标题（副本2）`、`原标题（副本3）`。
4. 创建后行为：不自动跳转到副本文档，保持当前上下文。

### 2.2 块级副本

1. 复制范围：仅复制当前块节点本身。
2. 插入位置：插入到当前块后方。
3. 编辑行为：光标定位到新副本块内，便于继续输入。
4. 历史行为：应支持一次 undo 回退整次复制。

## 3. 方案对比与决策

### 方案A：API驱动（推荐并采用）

1. 文档级副本走后端接口。
2. 块级副本走前端编辑器内本地操作，复用已有 autosave。

优点：边界清晰、改动最小、与当前架构一致；风险低。  
缺点：文档/块复制路径不同。

### 方案B：全后端驱动

文档与块复制都走后端。  
优点：审计集中。  
缺点：交互延迟高、实现复杂度不必要升高。

### 方案C：插件/Hook优先

先抽象 hook 事件再实现默认逻辑。  
优点：扩展性强。  
缺点：首版 YAGNI，成本偏高。

## 4. 架构与数据流设计

## 4.1 文档级副本（后端）

新增接口：

`POST /api/projects/:ownerType/:ownerKey/:projectKey/documents/:docId/duplicate`

请求体：首版可为空（预留 `title?: string`）。

处理流程：

1. 校验权限与文档存在性。
2. 读取源文档（title/content/parent/docType 等必要字段）。
3. 在同父级计算可用副本标题（带冲突递增）。
4. 复用现有 create 路径创建新文档并写入复制内容。
5. 返回新文档 detail（供前端即时更新树和缓存）。

错误语义：

1. `404`：源文档不存在。
2. `403`：无权限。
3. `409`：并发命名冲突（内部重试后仍冲突时）。

## 4.2 文档级副本（前端）

1. `apps/web/src/api/documents.ts` 增加 `duplicateDocument(projectKey, docId)`。
2. 文档页触发“创建副本”后调用接口。
3. 成功后最小状态同步：
   - 插入文档树对应父级列表
   - 更新 `documentsById` / cache
   - 可更新 recent edits（按现有策略）
4. 保持当前文档不切换。

## 4.3 块级副本（编辑器本地）

1. 在块菜单中新增“创建副本”动作（与“删除块/转换为”同级）。
2. 通过当前 `blockId` 找到顶层块 JSON。
3. 深拷贝节点并生成新块 ID（复用现有块 ID 机制）。
4. 在当前块后插入副本。
5. 触发现有 `onChange` + autosave 持久化。

## 5. 交互与可用性细节

1. 文档级副本成功：toast 提示“已创建副本”。
2. 文档级副本失败：toast 展示服务端错误信息。
3. 块级副本应兼容文本、列表、代码、媒体等通用块。
4. 块级副本入口文案统一：`创建副本`。

## 6. 测试策略

### 6.1 后端

1. 同父级创建副本并复制内容成功。
2. 标题冲突递增命名正确。
3. 源文档不存在返回 404。

### 6.2 前端页面

1. 点击文档“创建副本”后，文档树出现新节点。
2. 当前文档保持不切换。
3. 失败时错误提示正确。

### 6.3 编辑器（doc-editor）

1. 块菜单可见“创建副本”。
2. 复制后副本出现在源块后方。
3. 副本内容一致且 block id 不同。
4. undo 一次可回退复制。

## 7. 风险与回滚

1. 风险点：
   - 文档树局部状态更新遗漏导致显示不同步。
   - 块复制在复杂节点（如插件块）上的兼容性。
2. 回滚策略：
   - 前端隐藏入口按钮可快速止血。
   - 后端接口保留不影响主流程。

## 8. 非目标（首版不做）

1. 目录文档递归复制子树。
2. 块级“语义范围复制”（如可折叠标题下整段复制）。
3. 副本前自定义标题弹窗。

## 9. 结论

采用“文档后端复制 + 块前端本地复制”的混合方案，以最小实现成本交付明确价值，并保持后续扩展空间。
